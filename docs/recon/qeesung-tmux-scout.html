tmux‑scout → cc‑tmux

Does qeesung/tmux-scout — a 10-agent tmux session monitor — offer anything the just-shipped first-party apps/cc-tmux/ plugin doesn't? The two share a problem space but take opposite architectures: tmux-scout is external-JSON + watchdog daemon; cc-tmux is pane-options + daemon-free. That split decides every verdict.

Monitor & Skip — why the rest stays out

Monitor
MRU picker sort + pane-focus-in[9909] hook slot.
Coverage NONE, but flips cc-tmux's core sort (urgency-first → recency-first) with no caller demanding it — a /feature-sized design call. Keep the fixed-index hook-slot idiom in mind: idempotent, coexists with a user's own hook.
Skip 1
Watchdog daemon + single-writer Unix socket bridge.
FULL by design — cc-tmux spec makes pane options the sole state store, no daemon (rg socket/daemon → NONE). A daemon contradicts the invariant.
Skip 2
10-agent support (Codex, Gemini, Kimi, Copilot, OpenCode, Cursor, Hermes, Trae, Traex).
Out of scope — cc-tmux is a first-party Claude Code plugin.
Skip 3
Idempotent multi-CLI hook installer (backup + chain).
FULL via a different mechanism — cc-tmux self-registers through .claude-plugin/plugin.json + hooks/hooks.json, no config surgery.
Skip 4
Status-format placeholder DSL (@scout-status-format).
cc-tmux status already emits per-state styled counts; a format mini-DSL is over-engineering for a single-user plugin.

Implementation order — quickest win first

  1. Adapt 1 — fzf preview small — one fzf flag on the inbox popup; biggest UX lift, lowest cost. Pick the right pane without hopping.
  2. Adapt 2 — doctor subcommand small–medium — new cmd_doctor + parser entry; optionally called at the tail of the install script as a post-deploy self-check.
  3. Monitor — MRU sort — revisit only if urgency-first ordering proves annoying in daily use; needs a design decision, not a drop-in.

Before
No pane-state awareness in tmux at all. Manually check each parallel Claude pane (cmux workspaces).
After
Every pane's state tracked; cycle to the one needing you first; fzf inbox lists all waiting/idle across sessions; OS notify + terminal auto-focus on waiting.
  • Hook set more complete than prior tmux-agent-indicator finding: covers AskUserQuestion/ExitPlanMode/Elicitation/SessionEnd, 10s timeouts, real-transition guard against re-fired hooks.
  • Single source of truth: all views derive from one get_hop_panes() read of tmux pane options — can't disagree, auto-cleanup on pane close.
Layer: custom application (Python CLI + tmux shell entrypoint + hooks config) — not a skill/command/scripts-bin producer.
Landing: apps/cc-tmux/ — new top-level dir, tracked from day one (no vendoring/symlink).
Extend-before-create: nothing in if owns this domain; standalone is correct — Leo wants ONE combined plugin, not 3 separate installs.
Caller: every tmux session with parallel Claude panes (high, continuous). Effort: medium-large — this is the bulk of the planned scaffold.
⚠ Same cross-repo gate as the prior agent-indicator finding: hooks want ~/.claude/settings.json = ~/dev/cc symlink (cc-governed). One consolidated decision now instead of two.

Interesting, but opt-in even upstream and no caller in if yet — scripts/cmux-workspaces.sh is adjacent (launches workspaces) but solves a different problem (initial launch, not runtime dispatch to live panes). Build the state core first; revisit conductor once it has a real caller.

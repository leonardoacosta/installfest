# Network-reachability diagnostics — WHS-346 VNet / subnet / PE / DNS / NSG

> Load this file when diagnosing a WHS-346 VNet / subnet / private-endpoint / DNS / NSG reachability failure — a "subnet/endpoint unreachable at the SDN layer" verdict, before escalating to the network team. Skip when working on Graph, SNOW, or WHS pipelines/APIM/telemetry — those have their own references.

## Network-reachability diagnostics (WHS-346 VNet / MDP pool)

Before escalating "subnet/endpoint unreachable at the SDN layer" to the network team, rule out the cheap local causes — most "fabric defect" verdicts are a one-rule NSG fix.

- **An NSG `Deny` is a SILENT drop** — no TCP RST, no ICMP. A zero-hop `tracepath` or a `connect` timeout to an in-VNet destination is the NORMAL, EXPECTED signature of an NSG deny, NOT proof of an SDN/routing defect. Dump the SOURCE subnet's OUTBOUND NSG (and the dest subnet's inbound) first. For MDP / managed-pool agents the NIC lives in a Microsoft-owned sub (unqueryable) — that blind spot is exactly what makes an egress-NSG deny (e.g. the pool subnet's own `CustomDenyAllOutBound`) get misread as a fabric drop. Confirmed 2026-06-04.
- **A private-endpoint NIC's effective-route-table is NOT queryable** — `az network nic show-effective-route-table` on a PE NIC returns `NicMustBeAttachedToRunningVmToGetEffectiveRoutes`. But PE `/32 InterfaceEndpoint` routes propagate VNet-wide, so to check whether a PE's `/32` route exists, query the effective routes of ANY running-VM NIC in the same VNet (`az network nic list` → pick one with `.virtualMachine != null` → `show-effective-route-table`). MDP agent NICs are unqueryable (MS-owned sub) — use a customer IaaS/CPC VM NIC instead. Confirmed 2026-06-04.
- **A PE-only resource that NXDOMAINs / times-out is a DNS gap, not RBAC (`NXDOMAIN`/timeout != `403`).** The VNet's custom DNS (`10.234.0.36`) disables Azure's built-in `168.63.129.16` and lacks privatelink conditional forwarders, so VNet-integrated hosts and MDP-pool agents fail to resolve the `privatelink` FQDN to the PE IP. The signature is NXDOMAIN / TCP-connect-timeout, **never a 403** — a 403 means routing worked and the problem is RBAC. Four fix paths by context: (1) MDP/managed-pool builds — run `_templates/inject-pe-hosts.yml` before any KV/SQL/storage read; (2) cloudpc local data-plane — pin all PE IPs in cloudpc `/etc/hosts`; (3) App Service — set `WEBSITE_DNS_SERVER=168.63.129.16` explicitly (App Service does **NOT** inherit `dhcpOptions.dnsServers`); (4) durable — network team adds privatelink conditional forwarders to `10.234.0.36`. **Always log around KV/PE reads** — a `2>/dev/null` turns a DNS failure into a silent no-op (green pipeline, zero secrets, or a freshly-minted random key that diverges from canonical — SyncInscipherProxyKey produced APIM `b2d3eadb` != KV `43143fc2`).
- **A VNet with custom DNS disables Azure's built-in resolver.** Setting custom DNS (`10.234.0.36`) turns OFF `168.63.129.16` for every VM/agent in the VNet, so `privatelink.*` fails to resolve **unless the custom resolver carries conditional forwarders**. A name that won't resolve is a DNS problem, NOT an NSG/routing one — `inject-pe-hosts.yml` is the standing workaround.
- **`vnetRouteAllEnabled: false` routes ONLY RFC1918 through the VNet.** Public-DNS-resolved hosts (on-prem SQL via a public name, Kudu ZIP-deploy traffic, public storage endpoints) bypass the VNet, egress via the platform's public IPs, hit a storage/SQL firewall that doesn't allowlist those IPs, and TCP-time-out (or `403 AuthorizationFailure` on storage with `defaultAction: Deny`). Looks like a fabric problem, is a single app-setting. Fix = `vnetRouteAllEnabled: true` (all outbound → VNet → ExpressRoute/NVA). Most WHS satellites had `false` (CostCenter/DOC were exceptions); 7 Fireball functions 403'd on storage from exactly this.
- **Azure SQL connect-timeout (rc=258, ~15s, NOT a 403) = DATA-subnet NSG missing the Redirect port range.** Azure SQL defaults to the **Redirect** connection policy: client connects on `1433`, server redirects to an ephemeral port in `11000-11999`. The DATA-subnet *inbound* NSG must allow **both** ranges from the VNet-integration source subnet, or the connection silently times out. `SqlClient` records only the FQDN — the wire port is invisible at the app layer, so **verify NSG inbound rules, not app settings**; a Kudu `tcpping` on `1433` can succeed while the app times out. Check the **SQL dependency-success** metric, not request 200s. (TEST was missing `AllowVNetIntegrationSQLInbound` for `10.234.158.192/26`; GetSubmissionLock went 15s→0.5s after the add, 2026-06-08. DATA NSGs are network-team-curated — escalate the rule add, don't add it in Bicep.)
- **App can't reach an AD-domain-joined VM by FQDN (`*.Hull.com` / `*.hullco.com`) = `WEBSITE_DNS_SERVER=168.63.129.16` can't resolve the AD zone.** The "Hull on-prem" SQL/app servers (`W346PIPSSQL1<env>.Hull.com`, `W346ECCSQL01<env>.Hull.com`, `hull<env>-as.hullco.com`) are **Azure IaaS VMs *in* the WHS-346 VNet** (each env's DATA subnet `10.234.13{2/3/4}.x`), just AD-joined to the Hull domain. App Services/Functions pinned to Azure DNS (`WEBSITE_DNS_SERVER=168.63.129.16`, the privatelink-KV workaround) resolve `privatelink.*` but **NOT** the Hull AD zone — Azure DNS falls to PUBLIC DNS which returns garbage (`127.0.0.1` / a Rackspace IP), so the app connects to a bad IP → `(provider: TCP Provider) wait operation timed out / forcibly closed` (rc 258 / 10054), **0% success**. The corp DNS `10.234.0.36` resolves Hull but lacks privatelink forwarders, so you can't just switch onto it (KV breaks). **Diagnosis chain**: `TCP Provider` (not `SSL Provider`) rules out TLS; legacy app succeeds + canonical 0/N to the same host isolates it to the canonical DNS path; resolve the FQDN against `10.234.0.36` (returns the right `10.234.x`) vs what the app sees. **Fix (validated 0→100%, 2026-06-09)**: create **narrow per-host Azure Private DNS Zones** (one zone per FQDN, apex `@` A → the VM's IP) linked to `vnet-whs-346-centralus-dev` — Azure DNS then serves both privatelink AND Hull, **FQDNs stay in KV**, no IP-pinning, no `WEBSITE_DNS_SERVER` change, no network team. Per-host (not a broad `Hull.com` zone) avoids shadowing the AD domain for other teams. Project memory: `reference_fb_hull_dns_pips_connectivity`.
- **Mining real request payloads from legacy FB App Insights**: `requests` carry no bodies, but the FB functions LOG bodies into `traces` with `operation_Name=(none)` — so an `operation_Id` join MISSES them; **content-search wins** (`traces | where message has 'Request Body :' or startswith '{' or '"PolicyDetails"' / '"Transactions"'`). Reconstruct contracts for free from JSON deser errors: `... where m has 'Path: $.' | extend extract(@'Path: \$\.([A-Za-z0-9_]+)',1,m)`, cross-checked vs the `~/dev/fb` DTO. Legacy FB AIs (sub 979366b2): `fireball-stage-app-insights` d8192a61, `fireball-test-app-insights` 448f773a. Memory: `reference_ai_payload_mining_technique`.
- **Outbound NSG allow rules must enumerate each destination PE-subnet CIDR explicitly** — an aggregate/supernet that *should* cover the target silently drops it in the SDN fabric while still showing `provisioningState: Succeeded`. A 3-day "fabric defect" was a TEST pool egress NSG allowing only the TEST data CIDR (`.133`) and silently dropping STAGE data (`.134`). List every dest CIDR by hand. (Default `DenyAll` is priority 4096 — custom allows need a lower number.) Also: a PE with `privateEndpointNetworkPolicies: Disabled` has **no NSG applied to its traffic** — editing the subnet NSG won't change PE reachability.
- **VNet peering is NOT transitive; cross-region return paths need explicit allow.** A cross-region hub peering with `allowGatewayTransit: false` / `useRemoteGateways: false` silently drops traffic — the tell is a **~21s TCP SYN timeout** (cut at the handshake, no RST). A SYN timeout across regions = suspect peering/transit, not the destination.
- **The harness blocks writes to the Network RG independently of RBAC.** `az network vnet subnet update` on `RG-WHS-346-Network-*` is denied by an environment guardrail even when the identity *has* the RBAC — the Network RG is enterprise-managed (ws/CLAUDE.md, DO-NOT-MODIFY). Escalate any subnet/NSG/DNS change to the network team.
